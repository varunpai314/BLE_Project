# Integrate Firebase into Proximia Dashboard

This plan outlines replacing the current HTTP polling for `events` and `alerts` with real-time Firebase Firestore listeners, allowing the dashboard to fetch persistent history even across server restarts.

## Background Context
Currently, the `ble_dashboard` polls `http://192.168.29.76:8080/events` and `/alerts` every 3 seconds. The `ble_server` already pushes this data to the `proximia-argus` Firebase project's `events` and `alerts` collections. We will integrate Firebase directly into the Flutter Web dashboard to stream this data.

> [!IMPORTANT]
> **User Review Required: Firebase Configuration**
> Since this is a Flutter Web app, we need the Firebase Web configuration. Do you already have a `firebase_options.dart` file (generated by FlutterFire CLI)? If not, please run `flutterfire configure --project=proximia-argus` in the `ble_dashboard` directory to generate it, or provide the Web API key/App ID so we can initialize it manually.

## Proposed Changes

---

### `ble_dashboard` Dependencies

#### [MODIFY] pubspec.yaml
Add the necessary Firebase packages: 
- `firebase_core`
- `cloud_firestore`

---

### `ble_dashboard` Core Logic

#### [MODIFY] main.dart
1. **Initialize Firebase:** Update `void main()` to be `Future<void> main() async`, ensure `WidgetsFlutterBinding.ensureInitialized()`, and call `await Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform)`.
2. **Setup Firestore Listeners:** 
   - Remove `_fetchEvents()` and `_fetchAlerts()` HTTP request methods.
   - Create `_listenToEvents()` and `_listenToAlerts()` functions relying on `FirebaseFirestore.instance.collection(...).snapshots()`.
3. **Parse Documents:** Update the parsing logic to accept Firestore `DocumentSnapshot` data, mapping fields directly to the existing `ZoneEvent` and `PatientAlert` models.
4. **Current Patients:** We will continue polling `/patients` via HTTP as requested for the live demo, focused primarily on historical alerts for Firebase.
